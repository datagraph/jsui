1.1 Theoretical Foundation

Authentication, in the context of distributed systems, establishes a binding between a user agent and an identity known to the server. This chapter describes the authentication mechanisms supported by Dydra Studio and the procedures for their correct application.

1.2 Access Point

The canonical URL for the user interface is:

URL: https://dydra.com/ui/user

Unauthenticated requests to this URL result in the presentation of a login form. Authenticated requests proceed directly to the account view.

1.3 Authentication Procedure

The following algorithm describes the standard authentication process:

Step 1: Navigate to https://dydra.com/ui/user using a compatible web browser.

Figure 1.1: Authentication form with credential fields populated.

Step 2: Complete the form fields as follows:
  • Host: The server address, typically https://dydra.com
  • Login: Your assigned username
  • Password: Your authentication credential
Step 3: Submit the form by activating the Login button or pressing the Enter key.
Step 4: Upon successful validation, the system redirects to the account view. Failed authentication results in an error indication, requiring credential correction and resubmission.
Observation: The system supports concurrent authentication to multiple accounts within a single browser session. This capability proves useful when managing separate personal and organizational accounts.

1.4 Authentication Methods

The system supports two distinct authentication mechanisms:

1.4.1 Basic Authentication

The traditional username-password pair, transmitted via HTTP Basic Authentication headers. This method is appropriate for interactive browser sessions.

1.4.2 Bearer Token Authentication

For programmatic access, the system accepts bearer tokens in the Authorization header, formatted as: Bearer <token>. Tokens may be obtained from the account management interface (see Chapter 2).

The choice between methods depends on the access context. Human users typically employ Basic Authentication; automated systems and API clients use Bearer Tokens.

1.5 Session Persistence

1.5.1 What Is Preserved

The browser's local storage mechanism retains the username across sessions. This convenience feature reduces data entry upon subsequent visits.

Security Note: Passwords, tokens, and active session state are not persisted to local storage. Closing the browser tab terminates the session. This behavior is intentional: it limits exposure in shared computing environments.

1.5.2 Session Termination

To explicitly terminate a session:

  1. Locate the Logout control in the navigation area
  2. Activate the control
  3. The system returns to the unauthenticated login view
  4. The previously-used username remains available for convenience

1.6 Error Conditions

1.6.1 Authentication Failure

When authentication fails, verify the following:

  • Correct character case in the password (note keyboard state)
  • Correct host URL specification
  • Network connectivity to the server
  • Account existence and activation status

1.6.2 Error Message Interpretation

The system provides diagnostic messages upon failure:

  • "Invalid credentials" — Username or password mismatch
  • Network errors — Connectivity or server availability issues

Persistent difficulties should be reported to support@dydra.com with complete environmental details.

1.7 Post-Authentication Operations

Upon successful authentication, the following capabilities become available:

  • Account information review and modification (Chapter 2)
  • Repository creation and management (Chapter 3)
  • SPARQL view definition and execution (Chapter 4)
  • Reference material consultation (Appendices)
